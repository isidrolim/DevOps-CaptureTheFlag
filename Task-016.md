# Node Static Site Missing Build

## Scenario

A Node-powered static site server is running, but the generated deployment artifact directory is missing.

The server process is healthy, but the website cannot be served correctly because the static build output does not exist.

## Requirement

Generate the missing build artifacts so that:

```text
The generated static site exists
The JavaScript asset is available
The application returns the generated static website
```

## Initial State

The Node static server was already running.

However, the generated deployment directory:

```text
/srv/site/dist
```

did not exist.

Without the build artifacts, the running server had nothing to serve.

## Troubleshooting Path

```text
Understand the application architecture
↓
Inspect package.json
↓
Identify build and start scripts
↓
Inspect build script
↓
Determine expected build output
↓
Verify build artifacts are missing
↓
Generate build artifacts
↓
Validate generated website
↓
Validate generated JavaScript asset
```

## Understanding the Architecture

This application separates:

```text
Source Code

↓

Build Process

↓

Generated Static Website
```

The application source lives in:

```text
/srv/site/src
```

The build process is:

```text
node scripts/build.js
```

The generated deployment artifacts are written to:

```text
/srv/site/dist
```

The running static server serves files only from the generated `dist` directory.

If `dist` does not exist, the server is running but has nothing useful to serve.

## Verification Before Fix

Inspect the application scripts:

```bash
cat /srv/site/package.json
```

Result:

```json
"scripts": {
  "build": "node scripts/build.js",
  "start": "node scripts/static-server.js"
}
```

Inspect the build script:

```bash
cat /srv/site/scripts/build.js
```

Evidence from the script showed it generates:

```text
/srv/site/dist/index.html
/srv/site/dist/assets/app.js
```

Inspect the application directory:

```bash
ls -lah /srv/site
```

Result before the fix:

```text
package.json
scripts/
src/

dist/ was missing
```

## Systematic Elimination

```text
✓ Node static server is running
✓ Build script exists
✓ Start script exists
✓ Source files exist
✗ Generated deployment artifacts are missing
```

The first failed verification was:

```text
The generated deployment directory (/srv/site/dist) did not exist.
```

## First Finding

```text
The application was not missing source code.

The running server expected the generated build artifacts.

The build step had never been executed.
```

## Fix

Generate the deployment artifacts:

```bash
cd /srv/site
npm run build
```

The build process generated:

```text
/srv/site/dist
/srv/site/dist/index.html
/srv/site/dist/assets/app.js
```

## Validation

Confirm the generated deployment directory exists:

```bash
ls -lah /srv/site
```

Confirm generated assets exist:

```bash
ls -lah /srv/site/dist
ls -lah /srv/site/dist/assets
```

Because the Node application listens on port `3000` inside the container, validate from inside the container:

```bash
curl -i http://localhost:3000
```

Expected result:

```text
HTTP/1.1 200 OK

WEB-007 generated static site
```

Validate the JavaScript asset:

```bash
curl -i http://localhost:3000/assets/app.js
```

Expected result:

```text
HTTP/1.1 200 OK
```

After validation passes, click **Check Solution** and submit the generated flag.

## Lessons Learned

- Running an application is different from building an application.
- Static websites often require a build step before deployment.
- `package.json` clearly separates build scripts from runtime scripts.
- `npm run build` generates deployment artifacts.
- `npm start` serves the generated artifacts.
- A running service can still fail if required deployment artifacts are missing.
- Inside a Docker container, validate using the container's listening port (`3000`), not the host-mapped port (`32768`).

## Engineering Insight

One of the most important concepts in modern DevOps is understanding the difference between:

```text
Source Code

and

Deployment Artifacts
```

The source code is written by developers.

The deployment artifacts are generated by the build process.

Production servers normally serve only the generated artifacts.

A typical deployment pipeline looks like:

```text
Developer
↓

Source Code

↓

Build

↓

Generated Artifacts

↓

Deploy

↓

Run Application
```

If the build step is skipped:

```text
Source exists

↓

Artifacts missing

↓

Application starts

↓

Nothing useful to serve
```

This challenge demonstrates why CI/CD pipelines usually separate:

- Build
- Test
- Deploy
- Run

The server was healthy.

The application was healthy.

Only the deployment artifacts were missing.

Good engineers identify which lifecycle stage failed instead of assuming the application itself is broken.

Another important lesson from this challenge was understanding Docker networking.

Inside the container:

```text
localhost:3000
```

Outside the container:

```text
127.0.0.1:32768
```

Always validate using the correct network context.
